    07 · LLM 里的 RL：从 RLHF 到 PRM / Verifier / GRPO（通俗但不肤浅）¶

    这一章目标：你能用 RL 语言解释"LLM 对齐训练在干什么"，并理解为什么这几年很多团队从 PPO-RLHF 走向 DPO/GRPO/PRM 等路线。

    配套代码: code/rlhf/ 目录 - dpo_minimal.py — DPO 最小实现（可运行） - verifier_reward.py — Verifier 奖励系统（数学/代码/检索） - prm_toy.py — PRM 玩具版（过程奖励模型）

    1. LLM 训练的三段式（宏观地图）¶

    多数大模型的训练大概分三段：

    阶段 目的 数据 方法
    预训练 学语言基础 海量文本 Next-token 预测
    指令微调 (SFT) 学会按指令回答 指令 - 回答对 监督学习
    对齐 (Alignment) 符合人类偏好 偏好数据 RL / DPO / 其他

    RL 主要在第 3 段登场。

    2. RLHF：把"人类偏好"变成"奖励信号"¶

    RLHF（Reinforcement Learning from Human Feedback）的常见管线：

    2.1 收集偏好数据¶

    给同一个问题采样两个回答 A/B，让人标注"更喜欢哪个"。

    问题："什么是机器学习？"

回答 A: "机器学习是让计算机从数据中学习的科学。"  ← 人类更喜欢
回答 B: "机器学习是一种算法。"

    2.2 训练奖励模型（Reward Model, RM）¶

    用偏好数据训练一个模型：对回答打分，分数越高表示越符合人类偏好。

    奖励模型 (RM):
  输入：回答文本
  输出：分数 (0-1)

"机器学习是让计算机从数据中学习的科学。" → 0.85
"机器学习是一种算法。" → 0.42

    2.3 用 RL 训练语言模型（策略）¶

    把语言模型当策略 π： - 生成一段回答（动作序列） - RM 给一个整体奖励 - 用 PPO 等算法最大化奖励

    同时加 KL 约束：

    不要偏离参考模型太远（防止语言能力崩、胡言乱语）

    直觉：

    "朝着更讨人喜欢的方向变，但别一下子变得不像自己。"

    3. 为什么 PPO-RLHF 有工程痛点¶

    问题 描述
    在线采样成本高 每次训练都要生成新样本，计算量大
    系统复杂 需要 actor/critic/RM/参考模型 四个组件
    稳定性差 依赖很多技巧（reward scaling、KL 系数、batch、clip…）
    调参困难 超参数多，需要大量实验

    这促使很多团队寻找更简单的替代方案。

    4. DPO：不用显式跑 RL，也能把偏好"灌进策略"¶

    4.1 核心思想¶

    DPO（Direct Preference Optimization）把问题重写成一个更像"对比学习"的目标。

    直觉： - 你有偏好对 (chosen, rejected) - 你希望模型对 chosen 的概率更高，对 rejected 的概率更低 - 同时别偏离参考模型太远（相当于隐式 KL 约束）

    4.2 DPO 损失函数（简化版）¶

    L(θ) = -log[σ(β * (log(π_θ(chosen)/π_ref(chosen)) 
                  - log(π_θ(rejected)/π_ref(rejected))))]

其中：
- π_θ: 当前策略（要训练的模型）
- π_ref: 参考模型（固定，通常是 SFT 后的模型）
- β: KL 约束强度（越大越不允许偏离）
- σ: sigmoid 函数

    4.3 运行示例¶

    cd rl-tutorial-zero
source .venv/bin/activate
python3 code/rlhf/dpo_minimal.py

    输出示例： 

    ============================================================
DPO (Direct Preference Optimization) 最小实现
============================================================

配置:
  β (KL 约束) = 0.1
  学习率 = 0.01
  训练轮数 = 100

生成偏好数据：100 条样本

开始训练...
  Epoch   0: loss = 0.6932
  Epoch  10: loss = 0.6915
  Epoch  20: loss = 0.6882
  ...
  Epoch  99: loss = 0.6534

训练完成!

    4.4 DPO 优点¶

    优点 说明
    训练简单 不需要奖励模型，不需要在线采样
    稳定性好 直接优化，没有 PPO 的复杂技巧
    效果相当 多项研究表明 DPO 与 RLHF 效果接近

    4.5 DPO 局限¶

    • 仍然依赖偏好数据质量
    • 对复杂的"过程奖励/多步推理"可能需要额外设计

    5. RLAIF：用 AI 当标注员¶

    RLAIF（AI Feedback）用更强模型当裁判生成偏好或评分：

    用户问题 → 候选回答 → AI 裁判打分 → 训练数据

    优点： - 降成本（不用雇人标注） - 更快迭代

    风险： - 老师也会幻觉/偏见 - 需要高质量规则、抽检、人类锚点数据

    6. 过程监督：PRM 与 Verifier（近年很重要）¶

    很多任务（数学/代码/推理）的问题是： - 只看最终答案奖励太稀疏 - 模型可能"用语言骗奖励"（答案对但过程错）

    于是出现两类关键组件：

    6.1 PRM（Process Reward Model）¶

    对"中间步骤"打分： - 哪一步推理更合理 - 哪一步出现明显跳步/错误

    直觉：

    把稀疏的终点奖励，变成更密集的过程反馈。

    运行示例： 

    python3 code/rlhf/prm_toy.py

    输出示例： 

    ============================================================
PRM (Process Reward Model) 玩具版演示
============================================================

问题：解方程：2x + 3 = 7
最终答案：x = 2 (正确：✓)

过程评分：0.58
答案评分：1.00
综合评分：0.75

步骤分析:
  ⚠ 步骤 1 [assumption]: 设方程为 2x + 3 = 7... (分：0.60)
  ⚠ 步骤 2 [calculation]: 移项得 2x = 7 - 3... (分：0.60)
  ✓ 步骤 3 [calculation]: 计算得 2x = 4... (分：0.70)
  ⚠ 步骤 4 [inference]: 因此 x = 4 / 2 = 2... (分：0.60)
  ✗ 步骤 5 [conclusion]: 结论：x = 2... (分：0.40)

    6.2 Verifier / Unit-test Reward¶

    用可验证信号做奖励： - 数学：验算 - 代码：跑单元测试 - 检索：引用是否匹配

    这类奖励更"硬"，更能抑制幻觉。

    运行示例： 

    python3 code/rlhf/verifier_reward.py

    输出示例： 

    ============================================================
Verifier 奖励系统演示
============================================================

1. 数学题验证
  ✓ '答案是 6.28' → reward=1.00
  ✗ '我算出来是 5.0' → reward=0.80

2. 代码题验证
  ✓ reward=1.00 (所有测试通过)
  ✗ reward=0.00 (所有测试失败)

3. 检索题验证
  Citations: ['doc1', 'doc3', 'doc5']
    P=1.00, R=1.00, F1=1.00 → reward=1.00

    7. GRPO / Group-based 方法：减少对价值网络的依赖¶

    一些新路线用"同一问题多采样一组回答"的相对比较： - 对一组样本做归一化优势（相对排名） - 不一定需要训练一个复杂 critic

    直觉：

    同题多答，内部对比，谁更好就给谁更大正向更新。

    问题："什么是机器学习？"

采样 5 个回答:
  回答 1: ... → 得分 0.72
  回答 2: ... → 得分 0.85  ← 最好
  回答 3: ... → 得分 0.61
  回答 4: ... → 得分 0.79
  回答 5: ... → 得分 0.55

用组内相对排名计算优势，更新策略。

    8. 把 LLM 对齐映射回 RL 语言（你应该能说清楚）¶

    RL 概念 LLM 对齐中的对应
    状态 s prompt + 对话上下文
    动作 a 生成下一个 token（或一段文本）
    策略 π 语言模型本身
    奖励 r RM/PRM/Verifier 给的分数（可能只在结尾给）
    约束 KL 到参考模型、风格/安全规则

    9. 方法对比总结¶

    方法 复杂度 稳定性 数据需求 适用场景
    PPO-RLHF 高 中 偏好数据 + RM 训练 通用对齐
    DPO 低 高 偏好数据 通用对齐（推荐）
    RLAIF 中 中 AI 生成偏好 大规模/快速迭代
    PRM 中 高 过程标注数据 数学/代码/推理
    Verifier 低 高 可验证信号 代码/数学/检索
    GRPO 中 高 多样本生成 减少 critic 依赖
